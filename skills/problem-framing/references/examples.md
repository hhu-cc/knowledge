# Reframing Examples

## Example 1: Team efficiency

### Weak request

"The team is slow. We need more process."

### Better framing

- **Observed symptoms**: handoffs are delayed, decisions wait for approval, rework is common
- **Likely underlying problem**: decision rights are unclear and ownership is diffused
- **False requirement removed**: more process is not yet validated as the answer
- **Reframed problem**: reduce decision latency and rework by clarifying ownership boundaries and approval thresholds

## Example 2: Product growth

### Weak request

"Growth is flat. We need more features."

### Better framing

- **Observed symptoms**: activation is low, retention stalls early, feature adoption is fragmented
- **Likely underlying problem**: value proposition is not landing clearly in the core onboarding path
- **False requirement removed**: more features are not automatically more value
- **Reframed problem**: improve activation and early retention by making the core value easier to experience in the first session

## Example 3: Learning outcomes

### Weak request

"People are not learning well. We need more materials."

### Better framing

- **Observed symptoms**: completion rates are inconsistent, learners forget quickly, transfer to practice is weak
- **Likely underlying problem**: feedback loops and retrieval practice are too weak
- **False requirement removed**: more content volume is not the primary need
- **Reframed problem**: improve retention and transfer by strengthening active recall, feedback, and practice frequency

## Pattern to remember

When a request already contains a preferred solution, ask:

1. What symptom is this solution trying to address?
2. What evidence supports that this is the main cause?
3. What outcome would matter even if this solution were unavailable?
